# 密钥备份/恢复/轮换 SOP

- 文档版本：1.0
- 修订日期：2026-03-20
- 对应任务：OPS-003

## 1. 适用范围

适用于 `KEY_FILE`（默认 `./data/keys.json`）的备份、恢复与轮换。

## 2. 风险声明

1. `keys.json` 丢失且无备份将导致历史密文不可恢复。
2. `keys.json` 泄露将造成全部数据高风险暴露。
3. 密钥操作必须双人复核并留痕。

## 3. 备份流程（每日/每次变更后）

1. 停止对密钥文件的人工编辑。
2. 校验文件权限（生产建议 `400`）：

```bash
stat -c "%a %n" ./data/keys.json
```

3. 计算校验和并记录：

```bash
sha256sum ./data/keys.json
```

4. 采用加密介质离线备份（至少两份，异地一份）。
5. 记录备份时间、操作者、校验和、存放位置。

## 4. 恢复流程（灾备演练或故障恢复）

1. 停止应用进程。
2. 从可信备份恢复 `keys.json` 到 `KEY_FILE` 路径。
3. 恢复权限：

```bash
chmod 400 ./data/keys.json
```

4. 校验恢复文件校验和与备份记录一致。
5. 启动服务并执行验证：

```bash
CHECK_ONLY=1 ./scripts/start_local.sh
curl -s http://127.0.0.1:8000/healthz
```

6. 用已知样本执行一次查询，确认历史数据可解密。

## 5. 轮换流程（建议按季度）

说明：当前系统已支持多版本 key；历史密文可继续用旧版本解密，新写入使用激活版本。

1. 生成新版本 data/index key（32 bytes），追加到 `data_keys`、`index_keys`。
2. 更新：
   - `active_data_key_version`
   - `active_index_key_version`
3. 保存后设置权限 `400`。
4. 重启应用使新配置生效。
5. 回归验证：
   - 新导入记录可查询。
   - 历史记录可查询。
6. 保留旧 key 直到完成历史数据重加密（如后续实施）并完成验收。

## 6. 最小恢复演练清单（每月至少一次）

1. 在隔离环境用备份文件启动应用。
2. 验证登录、查询、导入三条主链路。
3. 核对审计日志完整性。
4. 输出演练记录并归档。
